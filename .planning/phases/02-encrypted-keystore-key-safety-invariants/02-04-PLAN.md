<objective>
Build the two startup safety rails that guard the keystore before any file is
written: `cloudsync.py` (SEC-04 — refuse a cloud-synced keystore dir by default,
downgrade to a loud warning only on explicit opt-in) and `passphrase.py`
(SEC-05 — confirm-on-create, no-echo, retry-on-mismatch, warn-on-short). Both
are pure/self-contained so `session.py` (02-05) can call them without circular
dependencies.

Purpose: A keystore silently replicated to a third-party cloud, or a mistyped
create-passphrase, quietly breaks the non-custodial guarantee. These rails make
the failure loud and deliberate. Resolves the two RESEARCH open questions:
(1) an empty `KEYSTORE_DIR` raises rather than silently defaulting (a silent
home-dir default could itself land under a cloud-synced path); (2) the override
is a function parameter `allow_cloud_sync` now — the interactive CLI flag /
Config wiring is deferred to Phase 7, mirroring the `--armed` opt-in philosophy.

Output: `bastion/keystore/cloudsync.py`, `bastion/keystore/passphrase.py`, and
their unit tests.
</objective>

  <behavior>
    - Matching entries: two identical getpass results → prompt_new_passphrase returns that passphrase.
    - Mismatch retry: first pair mismatches, second pair matches → returns the matched value (retry consumed, no raise); a third-attempt exhaustion (all mismatched) raises KeystoreError.
    - No echo: prompt_new_passphrase calls getpass.getpass (patched in the test), never builtins.input (assert input is not called).
    - Non-empty required: an empty passphrase entry is rejected (re-prompt or abort), never returned.
    - Short-passphrase warning: a matched but very short passphrase (below MIN_PASSPHRASE_WARN_LEN) returns the value AND emits a UserWarning (gentle, not a hard failure).
  </behavior>
  <action>
    Implement `prompt_new_passphrase(confirm_attempts=3)`: loop up to
    `confirm_attempts` times, each iteration reading two `getpass.getpass(...)`
    entries ("New keystore passphrase:" / "Confirm passphrase:"). Reject empty
    entries and mismatches by continuing the loop; on exhaustion raise
    `KeystoreError` (message-only, never echoes the entered value). On a match,

<objective>
Build `bastion/rpc/ws.py`: a persistent Solana WebSocket subscription client on the modern `websockets.asyncio` API, with an active heartbeat that catches silent drops, auto-reconnect, auto-resubscribe, and a "backfill needed" signal hook. Phase 1 delivers the mechanism and the hooks; the Phase 6 monitor wires the backfill signal into actual backfill dispatch.

Purpose: Silent WS drops are a documented Solana failure mode (PITFALLS.md #11) — `logsSubscribe` connections can go quiet with no close frame. Passive `onclose`/`onerror` listening misses this. Getting heartbeat-driven detection + resubscribe + backfill-signal right now is what makes the monitor's "never miss an event" guarantee (Phase 6) achievable.

Output (symbols this plan creates):
- `bastion/rpc/ws.py` — `ws_subscribe_logs(pubkey, callback, ...)`, `ws_subscribe_account(pubkey, callback, ...)`, an internal heartbeat/liveness timer, reconnect+resubscribe loop, and a backfill-needed (`on_gap`) callback hook
- `tests/unit/test_rpc_ws.py` — behavior suite against the local WS server fixture
</objective>

WS design notes (from RESEARCH.md Pattern 6 + Pitfall 2, authoritative):
- Use websockets.asyncio.client.connect() as an async iterator: `async for websocket in connect(uri, ping_interval=20, ping_timeout=20): ...` — it auto-reconnects with exponential backoff on transient errors. NEVER use websockets.legacy.
- ping_timeout alone does NOT catch a fully black-holed route (outgoing ping never leaves). Layer an INDEPENDENT liveness timer: track time since last ANY message; if it exceeds ~2x ping_interval, force-close the socket to trigger the outer reconnect.
- Subscriptions do NOT survive a reconnect — re-send every subscribe payload on each (re)connect.
- On an abnormal exit/reconnect, fire the backfill-needed signal BEFORE resuming live so the caller can backfill the gap.
